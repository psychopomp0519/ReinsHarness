# Systematic Debugging

체계적 디버깅 프로세스. 근본 원인을 찾기 전에 절대 수정하지 않는다.

<HARD-GATE>
## Iron Law: 근본 원인 없이 수정 금지

추측으로 코드를 수정하지 마라. 반드시 근본 원인을 찾은 후에만 수정한다.

| 이 생각이 들면 | 실제 해야 할 일 |
|--------------|---------------|
| "아마 이 부분이 문제일 거야" | 추측이다. 증거를 먼저 수집하라. |
| "이것만 고치면 될 것 같아" | 근본 원인을 확인했는가? 안 했으면 조사부터. |
| "빨리 고쳐야 해" | 급할수록 체계적으로. guess-and-check는 더 느리다. |
| "전에 비슷한 버그를 봤어" | 비슷해 보여도 다를 수 있다. 이 버그의 원인을 확인하라. |
</HARD-GATE>

## 4단계 디버깅 프로세스

### Phase 1: Investigate (조사)

근본 원인을 찾기 위한 정보 수집. 이 단계에서 코드를 수정하지 않는다.

**구체적 조사 단계:**

1. **에러 메시지 정확히 읽기**
   - 전체 에러 메시지와 스택 트레이스를 확인
   - 에러 타입, 발생 위치, 관련 변수 기록
   ```bash
   # 테스트 실패 시 전체 출력 확인
   # 로그 파일 확인
   # 브라우저 콘솔/네트워크 탭 확인
   ```

2. **콜 스택 추적**
   - 에러 발생 지점에서 역방향으로 호출 흐름 추적
   - 각 단계에서 데이터가 어떻게 변환되는지 확인
   - Grep으로 함수 호출 관계 파악

3. **최근 변경사항 확인**
   ```bash
   git log --oneline -10
   git diff HEAD~3
   ```
   - 버그가 언제부터 발생했는지 특정
   - 관련 파일의 최근 변경 이력 확인

4. **재현 조건 확인**
   - 항상 발생하는가, 간헐적인가?
   - 특정 입력/환경에서만 발생하는가?
   - 최소 재현 케이스를 만들 수 있는가?

### Phase 2: Analyze Patterns (패턴 분석)

수집된 정보에서 패턴을 찾는다.

- 에러가 발생하는 조건과 발생하지 않는 조건의 차이점
- 데이터 흐름에서 예상과 실제가 다른 지점
- 유사한 코드가 정상 동작하는 곳과의 비교
- 타이밍/순서 관련 이슈 여부 (race condition, 초기화 순서 등)

### Phase 3: Hypothesize + Test (가설 수립 및 검증)

1. 조사 결과를 바탕으로 **구체적인 가설**을 세운다
2. 가설을 **코드 수정 없이** 검증한다:
   - 로그/print 추가로 값 확인
   - 디버거로 중단점 설정
   - 단위 테스트로 특정 시나리오 검증
3. 가설이 틀리면 Phase 1로 돌아간다
4. 가설이 맞으면 Phase 4로 진행한다

### Phase 4: Implement Fix (수정 구현)

근본 원인이 확인된 후에만 수정한다.

1. **최소한의 수정** — 근본 원인만 정확히 수정
2. **회귀 테스트** — 수정 후 기존 테스트 전체 통과 확인
3. **엣지 케이스** — 유사한 패턴이 다른 곳에도 있는지 확인
4. **문서화** — 버그 원인과 수정 내용을 커밋 메시지에 명확히 기록

## 에스컬레이션 규칙

### 3+ 수정 실패 = 아키텍처 의심

동일 문제에 대해 3번 이상 수정을 시도했지만 실패한 경우:

1. **STOP** — 더 이상 같은 수준에서 수정하지 않는다
2. 문제가 코드 수준이 아니라 **설계/아키텍처 수준**일 가능성을 검토한다
3. 다음을 점검한다:
   - 잘못된 추상화 또는 책임 분리
   - 숨겨진 상태 의존성
   - 근본적인 데이터 모델 문제
   - 동시성/생명주기 문제
4. 아키텍처 이슈가 의심되면 → `/discuss`로 전환하여 설계를 논의한다
